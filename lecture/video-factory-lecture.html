Overview

概要

一言で言うと 「質問文字列を渡すと、スライド形式・音声ナレーション付き・図解入りのチュートリアル動画が自動生成され、 Webギャラリーに勝手に並ぶ」ツール。DevelopmentRAGEnvironment（既存・変更なし）のCloud RAGを情報源とし、 C++/Qtのヘッドレスレンダリングパイプラインで動画化する。

項目内容
入力トピック文字列 + 検索対象DB（dbKey）
出力.mp4（H.264映像 + AAC音声）+ Webダッシュボードへの自動公開
生成物の特徴スライド形式・SAPI音声ナレーション・Mermaid図解・コード例の音声解説
実行形態CLIバッチ実行 video_factory_cloudrag_poc.exe <topic> <dbKey>

設計ドキュメントとの関係

初期設計は docs/architecture/video-factory-design.md（Phase 0時点）にまとまっている。本講義資料はその後の実装（Phase 1〜2.5）で 実際にどう作られたかを追う内容。詳細な技術仕様は docs/technical-reference.md も参照。

Section 01

全体アーキテクチャ

3層構成
DevelopmentRAGEnvironment（既存・変更なし）
  └─ Cloud RAG GAS WebApp（Notion × Gemini）
        │ HTTPS POST /query
        ▼
Qt/C++ 動画ファクトリー（本リポジトリ engine/）
  ├─ CloudRagClient    ... HTTP+JSONクライアント
  ├─ NarrationEngine   ... Windows SAPI5 TTS
  ├─ スライド分割       ... splitIntoSlides→expandDiagramSlides→splitLongTextSlides
  ├─ SceneAssembler    ... QQuickRenderControl + QRhi（ヘッドレス描画）
  ├─ VideoEncoder      ... FFmpeg RAIIラッパー（H.264+AAC mux）
  └─ ManifestWriter    ... Webダッシュボードへ自動公開
        │
        ▼
Webダッシュボード（web/public/、静的サイトのみ）
  ├─ index.html   ... ギャラリー
  ├─ video.html   ... 詳細/プレビュー
  └─ random.html  ... ランダム連続再生（広告リール風）

設計上の一貫した方針

Faissのような専用ベクトルDBをC++側に埋め込まず、既存のCloud RAG HTTPブリッジをそのまま叩く。C++側は最後まで「HTTPクライアント」に徹し、 DevelopmentRAGEnvironment側のGASプロンプトにも一切変更を加えていない（Unity/Houdiniなど他の利用先に影響しないため）。

Section 02

エンジン: レンダリングパイプライン

1本の動画が出来上がるまでの流れ:

1
Cloud RAGへ質問 CloudRagClient

トピック文字列とdbKeyをPOST、回答本文(answer)と出典一覧(sources)を取得。

2
図解・コード説明の追加クエリ

回答にMermaidブロックが無ければ、図解+説明+コード説明を要求する2回目のクエリを送信(詳細は後述)。

3
ナレーション文生成 → SAPI TTS合成 NarrationEngine

Markdownをプレーンテキスト化し、Windows SAPI5で44.1kHzモノラルWAVを合成。実測音声長を取得。

4
動画の尺を決定

音声長 + 余韻1.5秒(最低4秒)。以前は固定6秒だった。

5
スライド分割 → フレームごとにレンダリング QQuickRenderControl

QRhi経由でオフスクリーンテクスチャへ描画、フレームごとに現在のスライド情報をQMLプロパティへセット。

6
FFmpegへpush → mux → ManifestWriterで公開 VideoEncoder

H.264映像+AAC音声をインターリーブしてmp4化。完了後、Webダッシュボードへ自動コピー。

VideoEncoderのRAII方針

FFmpegリソースラッパー型
AVFormatContextAVFormatContextPtr(カスタムデリータunique_ptr)
AVCodecContext(映像/音声 各1つ)AVCodecContextPtr
AVFrame / AVPacketAVFramePtr / AVPacketPtr
SwsContext(映像スケーリング) / SwrContext(音声リサンプリング)SwsContextPtr / SwrContextPtr

音声トラックはav_interleaved_write_frameのdtsベース自動整列に任せているため、映像→音声の順にすべて送っても正しくインターリーブされる。

Section 03

スライド分割ロジック(3段階)

「回答に##見出しがあるとは限らない」「1セクションが長すぎるとスクロールに頼りきりになる」という2つの実問題に対応するため、 3段階のパイプラインになっている。

スライド分割パイプライン
Cloud RAG回答(Markdown)
  │
  ├─ ① splitIntoSlides
  │     「## 見出し」でスライド分割(見出しが無ければ1枚にまとめる)
  │
  ├─ ② expandDiagramSlides
  │     「```mermaid」ブロックを検出→専用の図解スライドとして分離
  │     (mmdcでPNGにレンダリング)
  │
  └─ ③ splitLongTextSlides
        本文が200文字を超えるスライドを段落→文単位で強制再分割
        (2枚目以降は見出しに「(続き)」を付与)

バグ修正の経緯

当初は①だけだったため、見出しの無い回答（実際のCloud RAG回答は見出しが無いことが多い）が1枚の巨大スクロールカードになってしまい、 「スクロール形式の動画が多い」というフィードバックにつながった。③を追加することで、見出しの有無にかかわらず必ずダイジェスト（スライド）形式になることを保証した。

各スライドの表示時間は本文の文字数に比例配分(最低文字数フロアあり)。computeSlideStartFrames()がフレーム境界のルックアップテーブルを構築する。

Section 04

ナレーション(Windows SAPI5 TTS)

🔊 音声合成の仕組み
  • Windows標準SAPI5（ISpVoice/ISpStream）を素のCOM呼び出しで実装
  • ATL/sphelper.hは未使用（環境にATLが無いため）
  • 44.1kHz・モノラル・16bit PCM WAVを出力
  • ja-JPボイス（LCID 0x411）を優先選択、無ければデフォルトへフォールバック
⏱ 尺の自動調整
  • WAVファイルサイズから実際の音声長を逆算
  • 動画の尺 = 音声長 + 余韻1.5秒（最低4秒）
  • 以前の固定6秒から変更、長い回答でも短すぎる/間延びしない

選定理由: なぜクラウドTTSでなくSAPIか

ネットワーク不要・APIキー不要・完全オフラインで完結する点を優先した。品質は動作環境に入っているボイスに依存するため、 日本語ボイスが未インストールの環境では発音品質が下がる（将来的にはPhase 3のllama.cpp連携やクラウドTTSへの置き換えも検討）。

Section 05

Mermaid図解生成 & コード説明の補完

実際のCloud RAG回答（Gemini生成）には ```mermaid ブロックがほぼ含まれない （GAS側のプロンプトが図解生成を指示していないことを gas_cloud_rag.js の調査で確認済み）。 これに対応するため、LearningQt側だけで完結する追加クエリ機構を実装した。

追加クエリのシーケンス
① POST { query: topic, dbKey }              → 本文回答(answer)取得

   answerにmermaidブロックが無ければ:

② POST { query: "以下をMermaid図解+説明+コード説明で
             補足して: <answer>", dbKey }
   → 「図解説明: ...」+ ```mermaid + 「コード説明: ...」(登場順)を取得

③ 正規表現で抽出:
   ・図解説明   → answerに"## 図解"セクションとして合成
                 (既存のスライド分割が自動的に「説明文→図解」の2枚に分ける)
   ・コード説明[] → ナレーション生成時、コードフェンスの登場順に差し込み

この機構により、DevelopmentRAGEnvironment側（GASプロンプト）には一切変更を加えずに図解機能とナレーション品質向上を実現している。 Mermaidのレンダリング自体は mermaid-cli（npm、mmdcコマンド）をQProcess経由で呼び出し、 ブランドカラーに合わせたテーマ設定（engine/assets/mermaid_theme.json）でPNG化している。

ユーザーとの確認済み設計判断

論点決定理由

Section 07

実データで見つかったバグ修正

--mockではなく実際のCloud RAG認証情報・実データ(トピック「花火のパーティクルの作り方について教えてください」、 dbKey=houdini21)で生成した動画をユーザーが確認したところ、--mockのテストだけでは表面化しなかった 2件の品質バグが見つかった。

🔁 要点の重複
  • 症状: 左パネルの要点1・2に同一内容が2回表示、2回目は生の- マーカー付き
  • 原因: extractBullets()のフォールバック文分割が、既にリスト項目として拾った文を再度「新しい文」として拾ってしまっていた
  • 修正: 正規化(マーカー除去+空白除去)した内容をキーに重複排除。フォールバック側にも^[-*]\s+除去を追加
🔢 図解への出典番号混入
  • 症状: Mermaid図解に[1]由来の孤立した「1」だけのノードが混入
  • 原因: 出典番号付きの本文をそのままMermaid生成プロンプトに渡していた。Mermaid記法の[...]がノード構文と衝突
  • 修正: 共通ヘルパーstripCitationMarkers()を新設し、図解プロンプトへ渡す直前の3箇所すべてに適用

検証方法(1回目)

--mockフィクスチャは元々出典番号[1][2]を含む文面を持つため、この2件を再現できる。 修正後にビルドし直し、--mockで動画を再生成、ffmpegでフレームを間引き抽出して目視確認済み。

実データ第2ラウンド: 残っていた3件

上記2件の修正後、ユーザーが実際に"花火のパーティクルの作り方について教えてください" houdini21で再実行したところ、 --mockのテストパターンでは想定していなかった3件がさらに見つかった。

✂️ 要点の「先頭だけ重複」
  • 症状: 1行のリスト項目が「文A。- 文B。」のように2文を含む場合、要点1に全文、要点2に文Aの先頭を切り詰めたものが再表示
  • 原因: 前回の重複排除は完全一致のみ判定。この候補は既存要点の部分文字列に過ぎず一致せずすり抜けていた
  • 修正: フォールバック文分割の発動条件を「2件集まるまで」から「1件も無い時だけ」に変更(bullets.isEmpty())
📝 出典表記が数値限定だった
  • 症状: [参考: 過去Q&A]のような説明的な出典表記が要点・ナレーションに残る
  • 原因: stripCitationMarkers()の正規表現が\[\d+\](数字のみ)に限定されていた
  • 修正: \[[^\[\]]{1,60}\](角括弧内の任意の短いテキスト)に一般化。コードフェンスには適用されない箇所のみで使用
🔤 コンソールの文字化け
  • 症状: 実行ログの日本語行(例: topic=...)が文字化けして表示される
  • 原因: 実際のクエリ/生成内容には影響無し。logLine()が書き出すUTF-8バイト列を、コンソールの非UTF-8出力コードページが誤解釈していた(表示のみ)
  • 修正: main()冒頭でSetConsoleOutputCP(CP_UTF8)を呼び出すよう追加

副作用: ビルドエラー

<windows.h>を追加した際、そのmax/minマクロが既存のstd::max呼び出しと衝突し error C2589でビルド失敗。<windows.h>をインクルードする前に#define NOMINMAXを追加して解決した。

検証方法(2回目)

mockResponse()に①②の症状を再現する回帰テスト用セクションを追加し、ビルドし直して--mockで再生成、 フレーム抽出で要点重複ゼロ・出典表記の除去を確認済み。③はこのセッションにCloud RAG認証情報が無く実クエリでの再現ができなかったため、 修正は適用済みだが実データでの再確認はユーザー側で行う必要がある。

Section 09

実装で踏んだ落とし穴

症状原因対処
QQuickRenderControl::initialize()失敗QT_QPA_PLATFORM=offscreenはD3D11コンテキストを作れないデフォルトのwindowsプラットフォームのまま使う
実行時に即終了・無出力platforms/プラグインが実行ファイル横に無いCMake POST_BUILDで自動コピー
No functional TLS backend was foundtls/プラグイン・libssl-3-x64.dll未配置同様にPOST_BUILDでコピー
サムネイル保存失敗JPEGはQtの実行時プラグインが必要PNG形式に変更(標準搭載・プラグイン不要)
qDebug()の出力が見えないリダイレクト付きコンソールアプリでQtの既定ハンドラが機能しないことがあるstd::fprintf(stderr, ...)を直接使用
トピックを変えても古い動画に見える出力ファイル名が固定で毎回上書き実行時刻ベースのrunIdを全生成物に付与
見出し無し回答が全スクロール動画にsplitIntoSlidesのフォールバックが1枚にまとめていたsplitLongTextSlidesで強制再分割を追加
splitLongTextSlidesがコードフェンスを分断段落/文単位の再分割が```構文を認識しないコードフェンスを含むスライドは分割対象から除外
Visual Studioが「2022」→「18」へ自動更新、ビルドスクリプトが壊れるWindows/VS Installerの自動更新vswhere.exeで新パスを確認しスクリプト修正、CMakeキャッシュのみ再生成(vcpkg_installed/は保持)
std::max(...)がerror C2589でビルド失敗<windows.h>のmax/minマクロがstd::maxと衝突インクルード前に#define NOMINMAXを追加

共通する教訓

Qtのプラグイン（platforms/tls/imageformats）は「ビルドはできるが実行時に静かに壊れる」パターンが多い。 新しいQtモジュールを使い始めるたびに「実行時プラグインが必要か」を疑うクセをつけると事故が減る。

トークン消費量の可視化(推定)

実測値ではなく推定値

Cloud RAGのGASバックエンドは実際のトークン使用量を内部集計シートに記録しているが、クエリのレスポンスには含めていない。 共有バックエンド(Unity/Houdiniクライアントも依存)への変更は今回のスコープ外としたため、 クエリ・応答の文字数から概算した推定値を代わりに表示している。

engine/src/main_cloudrag.cppに文字数ベースの推定関数を追加し、メインクエリ・図解+コード説明follow-up・ スライド単位の図解follow-upの3箇所を合算して動画1本あたりの推定トークン消費量を算出する。

estimateTokens()
int estimateTokens(const QString& text) {
    // 日本語混在テキストのGemini系トークナイザでは概ね1.8文字/トークンが目安
    return static_cast<int>(std::round(text.size() / 1.8));
}

算出した合計値はmanifest.json/metadata.jsonのestimated_tokensフィールドに書き込まれ、 app.jsの共通ヘルパーtokenConsumptionHTML()が2箇所で描画する:

📊 index.html — ギャラリー
  • 累計推定トークン消費量(大きな数値)
  • 消費量が多い順に最大8本を横棒グラフでランキング表示
🎬 video.html — 詳細
  • その動画1本分の推定消費量のみ表示(ランキングではなく単一値)

estimated_tokensフィールドが無い(機能追加前に生成された)動画は集計から自動的に除外されるため、後方互換性の問題は無い。 このセッションはChrome拡張が使えなかったため、Node.jsでapp.jsを実行し実際のmanifest.jsonを渡して 出力HTMLを直接検証した(実ブラウザでのレイアウト確認は未実施)。
